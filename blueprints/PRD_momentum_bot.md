# PRD — 복합모멘텀 QQQM/GLDM 자동매매 봇

> **종목 표기 규약**: 전략은 나스닥100 지수와 금 현물 가격의 모멘텀을 비교한다. 실제 거래 종목은 저가·저보수 버전인 **QQQM(나스닥100)**과 **GLDM(금 현물)**으로 고정한다. 문서에서 "나스닥100"·"금"은 지수/자산 개념을, "QQQM"·"GLDM"은 실제 매매 티커를 가리킨다. QQQM/GLDM은 각각 QQQ/GLD와 동일 지수·동일 금 현물을 추종하므로 백테스트 신호는 동일하다(주당 가격이 낮아 정수 주문 자투리 현금이 감소하고 보수도 더 낮다).

> **문서 목적**: 이 PRD는 LLM 또는 개발자가 별도 추가 인터뷰 없이 프로젝트를 의도대로 구현할 수 있도록 작성되었다. 모든 핵심 로직, 폴백, 명령어, 배포·보안 결정이 포함되어 있다.

---

## Problem Statement

사용자는 백테스트로 검증한 **복합모멘텀 전략**(나스닥100과 금의 3·6·12개월 모멘텀을 비교해 매월 더 강한 자산으로 100% 전환, 둘 다 음수면 현금. 실제 거래 종목은 QQQM/GLDM)을 실제로 운용하고 싶다. 그러나:

- 전략의 신호 시점이 한국시간 한밤중(미국장 중)이라 매번 수동으로 주문하기 어렵다.
- 차트를 계속 들여다보는 데 시간을 쓰고 싶지 않다(기타 연습·앱 개발 등 다른 활동 우선).
- 그렇다고 사람 개입 없는 완전 자동에 전 재산을 맡기기엔 데이터 오류·시스템 장애·오발 주문이 불안하다.

즉, **"평소엔 알아서 돌아가되, 위험한 순간엔 사람이 개입할 수 있는"** 반자동 자동매매 시스템이 필요하다.

---

## Solution

한국투자증권(KIS) Open API + 텔레그램 봇 + Render Cron으로 구성된 자동매매 봇.

- **매월 말, 한국시간 자정 무렵(미국 정규장 중)** Cron이 깨어나 나스닥100/금 모멘텀을 계산하고 신호를 도출해 QQQM/GLDM으로 거래한다.
- **정상 신호는 사람 개입 없이 자동 주문**하고 결과를 텔레그램으로 사후 보고한다(기본 동작 = 자동).
- **이상치·대형 주문 등 위험 상황에서만** 텔레그램으로 y/n 승인을 요청한다(예외 동작 = 반자동).
- 사용자는 텔레그램 명령어(`/pause`, `/resume`, `/status`, `/signal`, `/log`, `/virtual`, `/real`, `/emergency-stop`, `/help`)로 봇을 통제한다. 봇은 **모의투자(virtual)와 실전(real) 두 모드**를 지원하며, 코드 수정 없이 명령으로 전환한다.
- 모든 폴백(데이터 결측·이상치, 미체결 확인, 부분 체결, 잔고 부족, 서버 장애, 휴장)이 정의되어 사고를 방지한다.

전략의 알파 원천은 "더 강한 자산 선택 + 폭락 회피"이며, 월 1회 결정이라 하루 이틀 지연돼도 성과 영향이 미미하다는 점이 모든 타이밍 관련 설계의 전제다.

## User Stories

### 전략 실행
1. As a 투자자, I want 매월 말 나스닥100과 금의 3·6·12개월 모멘텀이 자동 계산되기를, so that 내가 직접 수익률을 계산하지 않아도 된다.
2. As a 투자자, I want 두 자산 중 모멘텀 점수가 높은 쪽으로 전량 전환되기를, so that 전략 규칙이 기계적으로 집행된다.
3. As a 투자자, I want 나스닥100·금 둘 다 모멘텀이 음수일 때 전량 현금(매도 후 미보유) 상태가 되기를, so that 하락장에서 자산을 방어할 수 있다.
4. As a 투자자, I want 신호 계산이 미국 정규장 시간(한국시간 자정 무렵, 썸머타임 반영) 안에서 실행되기를, so that 주문이 실제로 체결될 수 있다.
5. As a 투자자, I want 정상 신호가 사람 개입 없이 자동 주문되기를, so that 한밤중에 깨어 승인하지 않아도 된다.
6. As a 투자자, I want 전환이 "기존 자산 전량 매도 → 신규 자산 매수"의 2-leg으로 처리되기를, so that 포지션이 명확히 한 자산에만 있게 된다.
7. As a 투자자, I want 신호가 직전 달과 동일하면(같은 자산 유지) 불필요한 매매가 발생하지 않기를, so that 거래비용이 낭비되지 않는다.

### 승인·알림 (반자동)
8. As a 투자자, I want 이상치가 감지되면 텔레그램으로 "현재값·평상시값"과 함께 y/n 승인을 요청받기를, so that 신뢰할 수 없는 데이터로 주문이 나가는 걸 막을 수 있다.
9. As a 투자자, I want 1회 주문 금액이 직전 평가금액의 110%를 초과하면 자동 주문 대신 y/n 승인을 요청받기를, so that 버그·데이터 오류로 인한 의도치 않은 대량 주문을 막을 수 있다.
10. As a 투자자, I want 자동 주문이 체결된 뒤 "전환 내역·체결가·수량·잔고 변화"를 텔레그램으로 통보받기를, so that 아침에 무슨 일이 있었는지 한눈에 파악한다.
11. As a 투자자, I want 승인 요청에 7일의 유효기간이 적용되기를, so that 한 사이클(한 달) 안에서 여유 있게 응답할 수 있다.
12. As a 투자자, I want 7일간 무응답 시 신호가 여전히 동일하면 한 번 더 승인을 요청받기를, so that 단순히 못 본 경우 기회를 놓치지 않는다.
13. As a 투자자, I want 재요청 후에도 신호가 달라졌거나 두 번째 7일간 무응답이면 해당 거래가 만료되기를, so that 오래된 신호로 뒤늦게 매매되지 않는다.
14. As a 투자자, I want 이상치 승인 요청이 미국장 마감 전까지 무응답이면 그날은 No-action으로 처리되고 다음 날 재시도되기를, so that 신뢰 불가 데이터로 자동 진행되지 않는다.

### 폴백 (장애·예외 처리)
15. As a 투자자, I want 모멘텀 계산에 필요한 과거 종가가 결측이면 직전 거래일 데이터로 거슬러 올라가 받아오기를, so that 일시적 결측으로 계산이 멈추지 않는다.
16. As a 투자자, I want 주문 전 미체결·잔고를 먼저 조회하기를, so that 네트워크 끊김 후 재시도 시 이중 주문이 발생하지 않는다.
17. As a 투자자, I want 2-leg 전환 중 한쪽(매도 또는 매수)만 체결되면 미완료 거래로 감지·알림되고 재시도되기를, so that 현금만 붕 뜬 상태로 방치되지 않는다.
18. As a 투자자, I want 이번 거래 후 잔고가 다음 거래(당시 금액 기준)에 부족할 것으로 판단되면 텔레그램 알림을 받기를, so that 미리 수동 환전으로 충전할 수 있다.
19. As a 투자자, I want 서버 장애나 KIS API 점검으로 폴백이 발생하면 텔레그램 알림을 받고 6시간 후 재시도되기를, so that 일시적 장애에 자동 복구된다.
20. As a 투자자, I want 휴장일 캘린더가 확인되면 활용하고, 확인 불가 시 2시간 간격 3회 재시도 후 모두 실패하면 휴장으로 간주되기를, so that 휴장일에 헛된 주문 시도가 반복되지 않는다.
21. As a 투자자, I want 폴백 재시도가 토큰을 재발급하지 않고 기존 유효 토큰을 재사용하기를, so that KIS의 토큰 발급 빈도 제한(1분당 1회)에 걸리지 않는다.

### 텔레그램 명령어
22. As a 투자자, I want `/pause`로 자동거래를 일시중지하기를, so that 시장 상황에 따라 봇을 멈출 수 있다.
23. As a 투자자, I want `/pause` 시 "진행 중 거래가 있으면 취소되고 일시정지됩니다 (y/n)" 재확인을 받기를, so that 의도치 않은 중지를 방지한다.
24. As a 투자자, I want `/pause` 확인 시 진행 중인 거래까지 취소되기를, so that 내가 멈추기로 한 순간 모든 거래 활동이 중단된다.
25. As a 투자자, I want 이미 일시중지 상태에서 `/pause`를 보내면 "이미 일시중지 상태"라고 안내받기를, so that 상태를 혼동하지 않는다.
26. As a 투자자, I want `/resume`으로 일시중지된 자동거래를 재개하기를, so that 다시 전략을 가동할 수 있다.
27. As a 투자자, I want 이미 작동 중에 `/resume`을 보내면 "이미 작동 중"이라고 안내받기를, so that 상태를 혼동하지 않는다.
28. As a 투자자, I want `/status`로 보유 주식 평가금액·수량·현재 잔고·다음 거래 잔고부족 여부·서버 상태를 텔레그램에 가독성 좋게 표시받기를, so that 현재 상태를 한눈에 본다.
29. As a 투자자, I want `/signal`로 "지금 시점 기준이라면 이번 신호가 무엇인지" 미리 보기를, so that 월말 전에 마음의 준비를 하고 전략이 살아있는지 확인한다.
30. As a 투자자, I want `/log N`으로 최근 N개 거래를 보고, N 없이 `/log`면 전체 거래를 최신순으로 요약받기를, so that 과거 거래를 추적한다.
31. As a 투자자, I want `/emergency-stop`으로 전량 매도 후 중지하기를, so that 급박한 상황에 즉시 위험을 제거한다.
32. As a 투자자, I want `/emergency-stop` 시 강력한 경고 확인 절차를 거치기를, so that 실수나 봇 탈취로 인한 오발 청산을 막는다.
33. As a 투자자, I want `/help`로 모든 명령어 도움말을 받기를, so that 사용법을 잊었을 때 참조한다.

### 모드 전환 (모의/실전)
33.1. As a 투자자, I want `/virtual`로 모의투자 모드로 전환하기를, so that 실제 자금 위험 없이 봇 동작을 검증할 수 있다.
33.2. As a 투자자, I want `/real`로 실전 모드 전환 시 챌린지-응답(확인코드 입력) 절차를 거치기를, so that 실수나 봇 탈취로 실계좌가 활성화되지 않는다.
33.3. As a 투자자, I want `/virtual` 전환은 간단한 y/n 확인만 거치기를, so that 안전한 방향(돈이 안 나가는) 전환은 번거롭지 않다.
33.4. As a 투자자, I want 모든 텔레그램 알림과 `/status` 최상단에 현재 모드([모의]/[실전])가 항상 표시되기를, so that "모의인 줄 알았는데 실전이었다"는 사고를 막는다.
33.5. As a 투자자, I want 현재 모드가 DB에 영속 저장되기를, so that 재배포·재시작 후에도 모드가 유지된다.
34. As a 투자자, I want 내 텔레그램 chat_id에서 온 명령만 처리되고 나머지는 무시되기를, so that 타인이 내 봇을 조작할 수 없다.

### emergency-stop 안전장치
35. As a 투자자, I want `/emergency-stop` 입력 시 "전량 청산됩니다. 정말? (y/n)" 2단계 확인을 받기를, so that 한 번의 실수로 청산되지 않는다.
36. As a 투자자, I want 확인 응답이 60초 내 없으면 청산이 자동 취소되기를, so that 실수로 누른 뒤 방치돼도 안전하다.
37. As a 투자자, I want 단순 y가 아니라 봇이 제시한 코드(또는 정해진 문구)를 그대로 입력해야 청산이 실행되기를, so that 봇이 탈취돼도 한 단계 더 막힌다.
38. As a 투자자, I want 청산 후 자동거래가 자동으로 pause 상태가 되기를, so that 청산했는데 다음 월말에 다시 매수되는 일이 없다.
39. As a 투자자, I want 청산 사유·시각·체결가가 `/log`에 별도 표시되기를, so that 비상 청산 기록을 추적한다.

### 운영·소액 운용
40. As a 투자자, I want 주문이 정수 수량으로 처리되기를(KIS API가 미국주식 소수점 미지원), so that API 제약에 맞게 동작한다.
41. As a 투자자, I want 정수 주문 후 남는 자투리 현금이 다음 거래까지 현금으로 보유되기를, so that 소액으로도 운용 가능하다.
42. As a 투자자, I want 잔고가 1주도 살 수 없는 수준이면 잔고부족 알림을 받기를, so that 충전 시점을 안다.
43. As a 투자자, I want 원화→달러 환전을 내가 임의 시점에 수동으로 하고 봇은 달러 기준 주문만 하기를, so that 환전 로직의 복잡도를 제거한다.
44. As a 투자자, I want 보유 포지션을 봇 메모리가 아니라 매번 KIS API로 실시간 조회하기를, so that 봇 상태와 실제 잔고가 어긋나지 않는다.
45. As a 투자자, I want 거래 로그·pause 플래그·미응답 신호 상태가 영속 저장되기를, so that 재배포·재시작 후에도 상태가 유지된다.

### 아키텍처·운영
46. As a 투자자, I want 텔레그램 명령을 받는 Web Service와 전략을 실행하는 Cron Job이 분리되기를, so that 월 1회 실행과 상시 명령 수신이 각자 역할에 맞게 동작한다.
47. As a 투자자, I want 두 서비스가 같은 레포·같은 Neon DB를 공유하되 직접 통신하지 않기를, so that 구조가 단순하고 상시 서버 추가가 필요 없다.
48. As a 투자자, I want `/pause`가 코드 푸시가 아니라 DB의 `is_paused` 갱신으로 처리되기를, so that 일시정지가 재배포 없이 즉시 반영되고 깃 히스토리가 오염되지 않는다.
49. As a 투자자, I want Cron이 기상 즉시 `is_paused`를 확인해 True면 알림만 보내고 종료하기를, so that 일시정지 중에 의도치 않은 거래가 발생하지 않는다.
50. As a 투자자, I want Web Service가 외부 keep-alive 핑으로 깨어있는 상태를 유지하기를, so that spin-down으로 텔레그램 명령을 놓치지 않는다.

## Implementation Decisions

### 아키텍처 개요 (★ 2-컴포넌트 구조로 확정)
- **실행 모델**: **같은 GitHub 레포를 공유하는 2개의 Render 서비스**로 구성한다.
  - **Web Service (상시 구동)**: 텔레그램 webhook을 24시간 수신해 명령을 즉시 처리한다(`/pause`, `/resume`, `/status`, `/signal`, `/log`, `/emergency-stop`, `/help`, y/n 응답). 상태 변경은 **Neon DB에 기록**한다. 실제 매매는 하지 않으며, 조회·상태변경·알림 응답만 담당한다.
  - **Cron Job (월 1회)**: 매월 말 자정 무렵 깨어나 전략을 실행한다. **기상 즉시 DB의 `is_paused`를 확인**하여, `True`면 거래를 건너뛰고 알림만 보낸 뒤 종료, `False`면 모멘텀 계산·주문을 수행한다.
- **두 서비스는 직접 통신하지 않는다.** Neon PostgreSQL이 유일한 공유 상태 저장소다(Web이 쓰고 Cron이 읽음). 이로써 별도의 폴링 크론이나 텔레그램 long-polling 워커가 불필요하다.
- **`is_paused` 의미 규약(혼동 방지)**: `is_paused = True` → 거래 중지(스킵 + 알림). `is_paused = False` → 정상 거래. 변수명은 "참이면 멈춘다"가 직관적으로 읽히도록 `is_paused`로 한다.
- **Web Service spin-down 대응 (★ 배포에서 변경: keep-alive 미채택 — KICKOFF 의사결정 로그 참조)**: Render 무료 web service는 15분 비활동 시 잠들고 재기동에 30~50초가 걸린다. 당초 UptimeRobot 5분 핑으로 상시 깨움을 유지하려 했으나, **무료 시간(~750h/월, 계정 내 free 서비스 공유) 절약을 위해 keep-alive를 쓰지 않고 spin-down을 허용**하기로 했다. 텔레그램 명령은 cold-start(~30~50초)로 깨어나 처리하며(유휴 후 첫 명령만 지연, 누락은 텔레그램 webhook 재시도로 방지), `/emergency-stop`도 이 지연을 수용한다. web service에는 `/healthcheck` 엔드포인트를 둔다(배포 헬스체크용).
- **실행 시각**: 매월 마지막 거래일, 미국 정규장 시간 내. 한국시간 기준 **썸머타임 반영 필수**(미국장 한국시간 23:30~06:00, 썸머타임 시 22:30~05:00). 자정~1시 타깃은 "미국장 개장 충격이 잦아든 시점"이라는 사용자 의도 반영.
- **기본 동작은 자동(레벨 3)**, 위험 상황에서만 텔레그램 승인(레벨 2)으로 전환되는 하이브리드.

### 핵심 모듈 (deep module 지향, 격리 테스트 가능하게 설계)

1. **MomentumEngine** (전략 두뇌, 순수 함수에 가깝게)
   - 입력: 나스닥100·금의 과거 종가 시계열(월말 기준점 포함). 실거래 종목 QQQM/GLDM의 가격을 그대로 사용해도 동일 지수라 신호 불변.
   - 처리: 각 자산의 3·6·12개월 수익률을 구해 **단순 평균**으로 모멘텀 점수 산출
   - 출력: 목표 자산(`NASDAQ` | `GOLD` | `CASH`) — 둘 다 점수 ≤ 0이면 `CASH`, 아니면 점수 높은 쪽. 실행 단계에서 `NASDAQ`→QQQM, `GOLD`→GLDM으로 매핑.
   - 외부 의존성 없음(데이터를 주입받음) → 단위 테스트 용이
   - 모멘텀 점수 정의(프로토타입에서 확정):
     ```
     score(asset) = mean( r_3m, r_6m, r_12m )
       where r_Nm = price[t] / price[t-N months] - 1   (월말 종가 기준)
     target = CASH                  if max(score_NASDAQ, score_GOLD) <= 0
            = argmax(score_NASDAQ, score_GOLD)   otherwise
     실행 매핑: NASDAQ → QQQM,  GOLD → GLDM,  CASH → 미보유(현금)
     ```

2. **MarketDataProvider** (KIS 시세 조회 + 결측/이상치 처리)
   - 월말 기준점들의 과거 종가 조회
   - 결측 폴백: 해당일 데이터 없으면 직전 거래일로 거슬러 올라가며 탐색
   - 이상치 감지: 전일 대비 비정상 변동(임계 정의 필요, 예: ±50%) 시 플래그 반환
   - BIL(현금성 지표)은 업로드 데이터에서 손상 이력이 있었으므로, **현금 수익률은 고정 일수익률 또는 미보유로 단순 처리**(주문 대상 아님)

3. **OrderExecutor** (KIS 주문 + 2-leg 전환 + 멱등성 + 모드 분기)
   - 거래 종목: `NASDAQ`→**QQQM**, `GOLD`→**GLDM**으로 고정 매핑
   - 2-leg 전환: 매도(기존 자산) → 매수(신규 자산) 순차 실행
   - **멱등성 보장**: 주문 전 미체결·체결·잔고를 조회해 "이미 처리됐는지" 확인 후 진행(중복 주문 방지)
   - 부분 체결 감지: 한쪽 leg만 성공 시 미완료 상태로 표시하고 재시도/알림
   - 정수 수량만(소수점 미지원). 매수 가능 수량 = floor(가용현금 / 현재가). QQQM/GLDM은 QQQ/GLD보다 주당 가격이 낮아 자투리 현금이 작음.
   - **모드 분기**: DB의 `trading_mode`(virtual|real)에 따라 KIS BASE URL·계좌를 분기. 코드 로직은 동일, 엔드포인트만 다름.

4. **PositionService** (실시간 잔고 — 상태 비저장)
   - 현재 보유 자산·수량·평가금액·현금을 **매번 KIS API로 실시간 조회**
   - 봇 내부에 포지션을 캐시하지 않음(실제 잔고를 단일 진실 원천으로 삼음)

5. **TokenManager** (인증 토큰 — 발급 빈도 제한 대응)
   - 실행 사이클 시작 시 토큰 1회 발급 → 그 사이클 내내 재사용
   - 토큰 유효기간 24시간. 폴백 재시도(6시간/2시간 후)는 **재발급 없이 기존 토큰 재사용**
   - 발급 빈도 제한(1분당 1회) 회피 설계
   - 앱키/앱시크릿은 환경변수에서 로드(코드·git에 평문 금지)

6. **ApprovalManager** (승인 상태 기계 — 7일 유효 + 재요청)
   - 승인 요청 상태: `PENDING` → (`APPROVED` | `REJECTED` | `EXPIRED` | `RE_REQUESTED`)
   - 일반 신호 승인은 7일 유효. 7일 무응답 + 동일 신호 → 1회 재요청. 재요청 후 신호 변경 또는 2차 7일 무응답 → `EXPIRED`
   - 이상치 승인은 별도 흐름: 미국장 마감 전 무응답 → 당일 No-action, 익일 재시도
   - 대형 주문(>110%) 승인도 이 매니저를 경유

7. **FallbackController** (장애 분기 오케스트레이션)
   - 데이터 결측/이상치, 미체결 확인, 부분 체결, 잔고 부족, 서버 장애(6h 재시도), 휴장(2h×3회 후 휴장 간주)
   - 각 폴백은 텔레그램 알림과 연계

7b. **ModeManager** (모의/실전 모드 — 상태 + 전환 안전장치)
   - DB의 `trading_mode`(`virtual` | `real`)를 단일 진실 원천으로 관리. 기본값은 안전한 `virtual`.
   - `/virtual` 전환: 간단한 y/n 확인(돈이 나가지 않는 방향이라 경량)
   - `/real` 전환: **emergency-stop과 동일한 챌린지-응답**(봇 제시 코드 입력) 필수. "실제 자금이 거래됩니다" 경고 포함.
   - OrderExecutor·PositionService·TokenManager가 이 값을 읽어 KIS 엔드포인트/계좌를 분기
   - 모든 알림·`/status`에 현재 모드 태그를 주입하는 단일 지점

8. **TelegramBot** (명령 수신·발신 + 인증 게이트) — **Web Service에서 구동**
   - 텔레그램 webhook으로 명령 수신(상시). 명령어: `/pause`, `/resume`, `/status`, `/signal`, `/log [N]`, `/virtual`, `/real`, `/emergency-stop`, `/help`
   - **인증 게이트**: 등록된 chat_id만 처리, 그 외 무시(모든 명령 공통 미들웨어)
   - 상태 변경 명령(`/pause`·`/resume`·`/virtual`·`/real`)은 Neon DB(`is_paused`/`trading_mode`)를 갱신만 하고, 실제 거래 차단·모드 분기는 Cron이 그 값을 읽어 수행
   - `/real`은 ModeManager의 챌린지-응답을 경유, `/virtual`은 y/n 확인
   - 조회 명령(`/status`·`/signal`·`/log`)은 KIS API·DB를 직접 조회해 즉시 응답
   - **모든 발신 메시지 머리에 현재 모드 태그(`[모의]`/`[실전]`)를 붙인다**(ModeManager 경유)
   - 메시지는 텔레그램 가독성에 맞춰 포매팅(아래 인터랙션 참조)
   - `/healthcheck` 엔드포인트 제공(UptimeRobot keep-alive 대상)

9. **StateStore** (영속 상태 — Neon PostgreSQL)
   - 저장 대상: **`is_paused` 플래그**, **`trading_mode`(virtual|real)**, pause 진행중 거래 취소 상태, 거래 로그(체결가·수량·손익·사유·시각·모드), 미응답 승인 상태, 마지막 신호
   - 저장 안 함: 현재 포지션(PositionService가 실시간 조회)
   - Web Service(쓰기)와 Cron Job(읽기)이 공유하는 유일한 통신 매개체

### 데이터 저장소 결정 (★ grill-me 이후 검증으로 변경됨)
- **1순위: Neon 무료 PostgreSQL** — 무기한 유지되는 무료 티어. 거래 로그를 장기간 쌓아야 하는 본 프로젝트에 적합.
- **Render 무료 PostgreSQL은 부적합**: 생성 후 **30일에 만료되고 데이터가 삭제**되며, 계정당 무료 DB 1개 제한. (단기 프로토타입에는 가능하나 운영 부적합)
- 어느 쪽이든 연결 문자열은 환경변수로 주입. 스키마는 마이그레이션 파일로 관리.

### 보안 결정
- 앱키/앱시크릿/DB 연결정보/텔레그램 봇 토큰 = **모두 환경변수**(Render Environment / Blueprint의 `envVars`). git·코드에 평문 금지.
- 거래 로그(체결가·수량·잔고)는 민감 금융정보 → **git 저장 금지**, DB에만 보관.
- 텔레그램 명령은 chat_id 화이트리스트로 제한.

### 배포 결정
- GitHub → Render **Blueprint(`render.yaml`)**로 배포(IaC). **하나의 레포에 web service와 cron job 두 서비스를 함께 정의**한다.
- DB는 Neon을 쓰므로 Blueprint에는 두 서비스만 정의하고 Neon 연결문자열을 두 서비스 공통 env로 주입.
- **`/pause` 등 상태 변경은 코드 변경/깃 푸시가 아니라 DB 갱신으로만 처리한다.** (깃 푸시 방식은 매번 재배포를 유발하고 깃 토큰 보관·커밋 오염·경합 문제가 있어 금지)
- (★ 배포에서 변경) web service 는 keep-alive 핑을 쓰지 않고 spin-down을 허용한다(무료 시간 절약 — 명령은 cold-start ~30~50초 지연 수용). cron 은 Render 에 무료 플랜이 없어 **starter(유료)** 로 띄운다.

### 주요 인터랙션 (텔레그램 메시지 형식 예)
- **사후 체결 보고**(자동 주문 후, 머리에 모드 태그):
  ```
  [실전] ✅ {YYYY-MM} 신호: QQQM → GLDM 전환 완료
     매도: QQQM {n}주 @ ${px} (체결)
     매수: GLDM {n}주 @ ${px} (체결)
     잔고: ${before} → ${after} (수수료·환차 ${delta})
     다음 거래 예상: {다음 월말}
  ```
- **이상치 승인 요청**:
  ```
  ⚠️ 모멘텀 이상치 발생 (현재 {now}, 평상시 {normal})
     그래도 거래하시겠습니까? (y/n)
  ```
- **pause 재확인**:
  ```
  현재 진행 중인 거래가 있다면 취소되고 일시정지됩니다.
  일시정지할까요? (y/n)
  ```
- **잔고 부족 알림**:
  ```
  ⚠️ 다음 거래 시 잔고 부족 예상 (필요 ${need}, 보유 ${have})
     수동 환전으로 충전해 주세요.
  ```
- **`/real` 실전 전환 (챌린지-응답)**:
  ```
  ⚠️ 실전 계좌로 전환합니다. 실제 자금이 거래됩니다.
     확인하려면 다음 코드를 그대로 입력하세요: {랜덤코드}
  ```
- **`/virtual` 모의 전환 (경량 확인)**:
  ```
  모의투자 모드로 전환할까요? (y/n)
  ```

## Testing Decisions

**좋은 테스트의 정의**: 구현 세부가 아니라 **외부로 관찰 가능한 행동**을 검증한다. 내부 함수 호출 순서나 사적 변수가 아니라 "이 입력에 이 출력/이 부수효과"를 본다.

**우선 테스트 대상(사용자가 테스트를 원하는지 확인 필요)**:
1. **MomentumEngine** ★ 최우선 — 순수 로직이라 테스트가 가장 쉽고 가치가 높다.
   - 케이스: 나스닥100 우위 / 금 우위 / 둘 다 음수(→CASH) / 동률 / 결측 보정된 입력
   - 업로드된 `historical_data.csv`의 특정 월말 구간으로 **기대 신호를 고정한 회귀 테스트** 작성 가능(백테스트에서 이미 신호열이 산출됨 → prior art). QQQ/GLD 데이터로 검증해도 QQQM/GLDM과 동일 지수라 신호 동일.
2. **OrderExecutor 멱등성** ★ — "이미 체결된 상태"를 주입했을 때 재주문이 발생하지 않는지.
3. **ApprovalManager 상태 기계** — 7일 만료/동일 신호 재요청/신호 변경 시 만료 분기.
4. **FallbackController** — 각 폴백 트리거 시 올바른 분기(재시도 간격·횟수·휴장 간주)와 알림 발생.
5. **OrderExecutor 부분 체결** — 한쪽 leg 실패 주입 시 미완료 표시·재시도.
6. **ModeManager 전환 안전장치** — `/real` 전환 시 잘못된/누락된 챌린지 코드는 거부되고 모드가 바뀌지 않는지, `/virtual`은 y/n로 전환되는지, 전환 결과가 DB에 영속되는지.

**테스트 격리**: KIS API·텔레그램·DB는 인터페이스로 추상화하여 목(mock)으로 대체. MomentumEngine은 외부 의존성이 없으므로 목 없이 순수 테스트.

## Out of Scope

- **환전 자동화**: 원화→달러 환전은 사용자가 수동으로 임의 시점에 수행. 봇은 달러 기준 주문만.
- **소수점(fractional) 주문**: KIS Open API 미지원으로 제외. 정수 주문 + 자투리 현금.
- **ISA·연금저축 계좌 연동**: API 주문은 일반 위탁계좌 기준. 본 봇은 일반계좌(20% 새틀라이트 자금)에만 적용.
- **세금 처리·신고 자동화**: 범위 외(`/log`로 내역 추적만 제공).
- **다중 사용자/타인 공유**: 사용자 본인 단독 사용. chat_id 화이트리스트 1인.
- **SPY 등 추가 자산으로의 전략 확장**: 본 PRD는 나스닥100(QQQM)/금(GLDM) 2자산 + 현금에 한정.
- **거래 종목 선택(QQQ↔QQQM 등) 명령어**: 종목은 QQQM/GLDM으로 고정. 동일 지수 종목 간 전환 옵션(`/mode`)은 전환 비용·복잡도 대비 이익이 없어 제외(YAGNI).
- **완전 무인 청산 판단**: 비상 청산은 사용자가 `/emergency-stop`으로 명령(자동 청산 로직 없음).
- **백테스트 기능 내장**: 별도. 본 봇은 라이브 실행 전용(전략은 이미 검증됨).

## Further Notes

- **전략 성과 전제**: 백테스트(2006–2026 실데이터, QQQ/GLD 기준)에서 복합모멘텀은 CAGR 17.24%, MDD -32.3%로 SPY B&H(9.58%, -56.5%)를 상회했다. 알파 원천은 ① 나스닥100/금이 SPY보다 강했던 자산 선택, ② 2008·2022 폭락 회피. **미래에 나스닥100이 장기 침체하면 우위는 재현되지 않을 수 있으나, 그 경우에도 모멘텀 구조가 금/현금으로 피신**하도록 설계됨. 실거래 종목 QQQM/GLDM은 각각 QQQ/GLD와 동일 지수·동일 금 현물을 추종하므로 백테스트 결과가 그대로 적용되며, 보수가 더 낮아 장기 순수익은 오히려 소폭 유리. 배당 미반영으로 SPY와 실제 격차는 ~1%p 좁아질 수 있으나 결론 불변.
- **종목 선택 근거(QQQM/GLDM)**: QQQM은 QQQ와 동일 나스닥100 추종, 보수 0.15% vs 0.20%, 주당 가격 약 $291 vs $708. GLDM은 GLD와 동일 금 현물 추종, 보수 0.10% vs 0.40%, 1년 수익률·MDD 거의 동일. 월 1회 거래라 QQQM/GLDM의 낮은 유동성은 무관. 정수 주문 자투리 현금이 절반 이하로 감소해 소액 운용에 유리.
- **모드 전환(virtual/real)**: KIS는 실전/모의가 BASE URL만 다르고 API 구조 동일 → 같은 코드가 두 모드에서 동작. `trading_mode`를 DB에 두고 `/virtual`·`/real`로 전환. 신규 기능·폴백 수정은 `virtual`로 먼저 1~2 사이클 검증 후 `/real` 승격. **모의투자 계좌는 주기적 초기화·만료될 수 있어 검증용으로만 사용**(장기 무인 운용은 real). `/real` 전환은 챌린지-응답으로 오발 방지.
- **타이밍 지연 허용**: 월 1회 결정 전략이므로 하루~이틀 체결 지연은 성과에 거의 영향 없음. 모든 폴백의 "No-action 후 재시도" 기조는 이 전제에 근거한다.
- **KIS 토큰 제약**: 접근토큰 24시간 유효, 발급 빈도 제한(1분당 1회). 사이클당 1회 발급·재사용 원칙 준수.
- **KIS 호출 제한**: 2026년 3월부터 신규 고객 초당 호출 제한 적용. 월 1회·저빈도라 정상 운영엔 무관하나 폴백 재시도에서 연속 호출 자제.
- **썸머타임**: 미국 DST 전환에 따라 실행 시각이 한국시간으로 1시간 이동. Cron 스케줄을 DST 인식하도록 구성.
