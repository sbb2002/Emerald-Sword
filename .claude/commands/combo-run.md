# Combo Run

이슈 1개를 pre-flight → 작업 → syntax-gate → post-patch 루프로 실행하는 오케스트레이터 스킬.

## 사용법

```
/combo-run           — 전체 이슈 순서대로 루프 실행
/combo-run N         — 이슈 N번 단일 실행 후 종료
/combo-run --list    — 대기 중인 이슈 목록 출력 후 선택
```

---

## 실행 흐름 (이슈 1개 기준)

```
① 이슈 선택 및 내용 파악
        ↓
② pre-flight-check  — 건드릴 파일 목록 선언 + 사용자 확인
        ↓ (승인)
③ 작업 실행         — 해당 이슈 구현 (중대 결정 감지 시 즉시 중단 → 사용자 확인)
        ↓
④ syntax-gate       — 문법 검사 → 오류 시 수정 루프
        ↓ (통과)
⑤ post-patch-diff   — 누락 파일 검증 → 누락 시 수정 루프
        ↓ (완료)
⑥ 의사결정 로그 기록
        ↓
⑦ 완료 처리
```

---

## 모드별 진입 방식

### `/combo-run` (루프 모드)
1. 대기 중인 이슈 목록을 우선순위 순으로 출력한다.
2. 가장 우선순위 높은 이슈부터 자동으로 시작한다.
3. 이슈 완료 후 → 다음 이슈로 자동 진행한다.
4. 모든 이슈 완료 시 → 루프 종료 메시지 출력.

### `/combo-run N` (단일 모드)
1. 이슈 N번의 내용을 불러와 바로 실행한다.
2. 완료 후 → 종료. 다음 이슈로 넘어가지 않는다.

### `/combo-run --list` (선택 모드)
1. 대기 중인 이슈 전체를 표로 출력한다:
   ```
   | # | 이슈 제목 | 타입 | 블로커 | 상태 |
   |---|-----------|------|--------|------|
   | 1 | 로그인 UI | AFK  | 없음   | 대기 |
   | 2 | API 연동  | AFK  | #1     | 대기 |
   ```
2. 사용자가 번호를 선택하면 해당 이슈 단일 실행.
3. 완료 후 → 종료.

## 단계별 진행 규칙

### ① 이슈 파악
- 이슈 제목, 구현 내용, 인수 조건(Acceptance criteria), 블로커를 확인한다.
- 블로커가 미완료 상태면 사용자에게 알리고 다른 이슈를 제안한다.

### ② pre-flight-check
- pre-flight-check 스킬을 그대로 실행한다.
- 파일 목록 확인 후 사용자 승인 대기.
- 승인 전까지 ③으로 넘어가지 않는다.

### ③ 작업 실행
- 승인된 파일 목록 범위 안에서 이슈를 구현한다.
- 범위 외 파일이 필요해지면 → 즉시 멈추고 pre-flight 목록 갱신 후 재확인.
- **중대 결정 감지 시 즉시 멈추고 사용자에게 확인** (아래 "중대 결정 감지" 참조).
- **Kill Condition 감지 시 경고 알림 후 진행 여부 확인** (아래 "Kill Condition 경고" 참조).

### ④ syntax-gate (오류 수정 루프)
- syntax-gate 스킬을 실행한다.
- ✅ 통과 → ⑤로 진행.
- ⚠️ 경고 → 사용자에게 보고 후 진행 여부 확인.
- ❌ 차단 → 즉시 수정 후 해당 항목 재검사. 통과할 때까지 반복.
- 수정 로그 형식:
  ```
  [syntax-gate 오류 로그]
  발생: useCallback import 누락 (Button.tsx line 3)
  수정: import { useCallback } from 'react' 추가
  재검사: ✅ 통과
  ```

### ⑤ post-patch-diff (누락 수정 루프)
- post-patch-diff 스킬을 실행한다.
- ✅ 완료 → ⑥으로 진행.
- ❌ 누락 → 즉시 해당 파일 패치 후 해당 항목 재검증. 완료할 때까지 반복.
- ⚠️ 예상 외 변경 → 사용자에게 보고 후 의도 확인.
- 수정 로그 형식:
  ```
  [post-patch 누락 로그]
  누락: src/types/index.ts (ButtonProps 미반영)
  수정: ButtonProps 타입 추가
  재검증: ✅ 완료
  ```

### ⑥ 의사결정 로그 기록

이슈 완료 직후, KICKOFF.md의 "의사결정 로그" 섹션에 아래 형식으로 추가한다:

```
[날짜] 이슈 #N — [결정 내용] / [이유] / [포기한 대안]
```

예시:
```
[2026-06-09] 이슈 #2 — JWT 대신 세션 쿠키 채택 / 모바일 환경에서 토큰 저장 리스크 / JWT stateless 방식 포기
[2026-06-09] 이슈 #3 — 실시간 알림 WebSocket → polling으로 변경 / 서버 비용 절감 / SSE도 검토했으나 브라우저 호환성 이슈
```

기록 대상:
- 구현 중 내린 아키텍처 결정
- 포기한 접근 방식
- 중대 결정에서 사용자가 선택한 내용
- Kill Condition 경고에서 사용자가 내린 판단

기록할 내용이 없으면 (단순 UI 수정 등) 생략해도 된다.

### ⑦ 완료 처리

**단일/선택 모드** (`/combo-run N` 또는 `--list`):
```
## 이슈 #N 완료

✅ pre-flight  통과
✅ syntax-gate 통과
✅ post-patch  완료
오류 수정: N건 (있을 경우)
의사결정 로그: N건 기록

/combo-run --list 로 다음 이슈를 선택하거나
/handoff 로 세션을 마무리할 수 있어요.
```

**루프 모드** (`/combo-run`):
```
## 이슈 #N 완료 → 이슈 #N+1 시작합니다.
```

모든 이슈 완료 시:
```
## 전체 이슈 완료 🎉

완료: N개 / 전체: N개
오류 수정 총 건수: N건
의사결정 로그: N건 기록

/handoff 또는 /give-me-receipt 로 마무리할 수 있어요.
```

## 중대 결정 감지

작업 중 아래 항목에 해당하는 구현이 필요해지면 **즉시 멈추고** 사용자에게 확인한다.
Claude가 임의로 결정하지 않는다.

### 보안 관련
- 인증 방식 선택 (JWT / 세션 / OAuth / 패스워드리스)
- 토큰 저장 위치 (localStorage / httpOnly 쿠키 / 메모리)
- 암호화 알고리즘 또는 해싱 방식 선택
- API 키·시크릿 관리 방식
- CORS 정책 설정
- 권한(Role) 모델 설계

### 앱 수명주기 관련
- 데이터베이스 스키마 변경 (마이그레이션 필요)
- 외부 API 또는 서드파티 서비스 도입
- 배포 방식 변경 (서버리스 ↔ 컨테이너 등)
- 캐싱 전략 선택
- 상태 관리 라이브러리 도입 또는 교체
- 모노레포 / 멀티레포 구조 변경

### 확인 형식
```
⚠️ 중대 결정 필요 — 작업을 잠시 멈춥니다.

이슈 #N 구현 중 아래 결정이 필요합니다:

결정 사항: [구체적인 선택지]
영향 범위: [어떤 부분에 영향을 미치는지]
선택지:
  A. [옵션 A] — [장단점]
  B. [옵션 B] — [장단점]
권장: [Claude의 권장 옵션과 이유]

어떻게 진행할까요?
```

사용자 선택 후 → 의사결정 로그에 기록하고 작업 재개.

## Kill Condition 경고

작업 중 KICKOFF.md의 "Kill Condition" 항목에 해당하는 상황이 감지되면
**즉시 멈추고** 아래 형식으로 경고한다.

```
🚨 Kill Condition 감지 — 중단 여부를 결정해주세요.

KICKOFF.md에 명시된 중단 조건이 충족될 수 있습니다:

감지된 조건: [KICKOFF.md의 Kill Condition 내용]
현재 상황: [왜 이 조건에 해당하는지 구체적 설명]
영향: [계속 진행할 경우 발생할 수 있는 문제]

선택:
  계속  — 이 조건을 무시하고 진행합니다 (이유를 로그에 남깁니다)
  중단  — 현재까지의 작업을 보존하고 프로젝트를 중단합니다
  재검토 — Kill Condition 자체를 수정한 후 진행합니다

어떻게 하시겠어요?
```

사용자 선택에 따라:
- **계속** → 사유를 의사결정 로그에 기록하고 작업 재개
- **중단** → 현재까지 작업 보존, `/handoff` 실행 권장
- **재검토** → KICKOFF.md의 Kill Condition 수정 후 재개

---

## 오류 수정 원칙

- 오류는 **즉시 수정**한다 — 다음 단계로 미루지 않는다.
- 수정 시도는 **최대 3회**. 3회 내 해결 안 되면 사용자에게 보고하고 판단을 위임한다.
- 모든 수정 내역은 로그로 남긴다.
- ⚠️ 경고는 통과로 처리하되, 완료 리포트에 포함시킨다.
