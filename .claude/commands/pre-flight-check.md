# Pre-flight Check

코드 작업 실행 전 파일 범위와 변경 계획을 명시적으로 선언하는 스킬.

## 목적

패치 누락·문법 오류의 가장 흔한 원인은 **"어떤 파일을 건드려야 하는지 합의가 없었던 것"**이다.
실행 전에 범위를 선언하면 누락을 사전에 막고, 작업 후 검증의 기준선이 생긴다.

---

## 실행 순서

### 1. 작업 의도 파악

사용자의 요청에서 다음을 추출한다:
- **무엇을** 구현/수정/삭제하는가
- **어느 레이어**에 영향이 있는가 (UI, 로직, 데이터, 설정 등)

### 2. 영향 파일 목록 선언

아래 형식으로 작업 전에 반드시 제시한다:

```
## Pre-flight Check

### 건드릴 파일
| # | 파일 경로 | 변경 유형 | 변경 이유 |
|---|-----------|-----------|-----------|
| 1 | src/components/Button.tsx | 수정 | 클릭 핸들러 추가 |
| 2 | src/styles/button.css | 수정 | hover 스타일 적용 |
| 3 | src/types/index.ts | 수정 | ButtonProps 타입 확장 |

### 건드리지 않을 파일
- src/pages/Home.tsx — Button을 사용하지만 인터페이스 변경 없음

### 전제 조건
- [ ] Button 컴포넌트가 이미 존재하는가?
- [ ] 타입 파일 경로가 맞는가?

작업을 시작할까요?
```

### 3. 사용자 확인 후 진행

- 사용자가 **확인**하면 → 작업 시작
- 파일 목록이 **틀렸다면** → 수정 후 재확인
- **전제 조건**이 불충족이면 → 해소 후 진행

---

## 변경 유형 분류

| 유형 | 의미 |
|------|------|
| 신규 | 새 파일 생성 |
| 수정 | 기존 파일 내용 변경 |
| 삭제 | 파일 제거 |
| 이동 | 경로 변경 (구 경로 → 신 경로) |

---

## 규칙

- 파일이 **0개**일 수 없다 — 코드 작업이면 반드시 1개 이상
- **"아마도"** 바뀔 파일도 포함한다 (확실하지 않으면 `?` 표시)
- 작업 도중 예상 외 파일이 필요해지면 → 멈추고 목록 갱신 후 재확인
- Claude.ai 아티팩트는 파일 경로 대신 **컴포넌트명**으로 표기

---

## 빠른 모드 (`/pre-flight quick`)

긴 테이블이 불필요한 소규모 작업용:

```
## Pre-flight (quick)
건드릴 것: Button.tsx (클릭 핸들러), button.css (hover)
건드리지 않을 것: 나머지 컴포넌트
진행할까요?
```
